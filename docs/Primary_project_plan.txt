### Prototype Architecture — Directly Derived from Both Papers


This is the minimal viable system that faithfully reflects both references:


**Phase 1 — Replicate Madanian et al. on EMO-DB + RAVDESS:**
```
Audio (.wav, 16kHz mono)
    │
    └──► PyAudioAnalysis: extract 34 features
         (MFCC, Energy, ZCR, Entropy of Energy, 
          Chroma, Spectral Centroid, Spectral Entropy)
              │
              ├── SVM (C = 0.5–20, overlapping window 50ms/25ms step)
              ├── Random Forest (N=100)
              ├── Gradient Boosting (100 stages)
              └── Extra Trees (N=100)
                        │
                   Emotion Label (7-class)
```


**Phase 2 — Add Yurtay et al.'s insight (collapse + text branch):**


Yurtay's key finding was that **3-class sentiment (Positive/Negative/Neutral) is more actionable than 7-class emotion** for a call center operator, and that **text carries 75% of the signal**. Map your EMO-DB labels:
- Positive → happiness
- Negative → anger, disgust, fear, boredom, sadness  
- Neutral → neutral


Then add Whisper (for ASR transcription) → simple BERT-based sentiment classifier on the transcript. Fuse at 75/25 ratio as Yurtay demonstrated.


**Phase 3 — Frame it as system design (the actual thesis question):**


This is what separates a prototype from a demo. Add a thin layer that shows *where in the call workflow* the emotion signal would be consumed — for Allianz this means: real-time alert to the agent dashboard when the customer sentiment turns negative. Even a mock Gradio UI showing "Customer: NEGATIVE ↓ | Agent: NEUTRAL" is enough to make the system design argument concrete to the mentor.
